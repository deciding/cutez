## Top-Down Model

`modal/01_dense_gemm.py` is a benchmark dispatcher. It does not implement the GEMM kernel itself. Instead, it builds a Modal image, mounts the `blackwell` sources into the container, selects one or more benchmark targets through `RUN_TESTS`, and then calls the corresponding Python entrypoint inside `run_dense_gemm()`.

For `dense_gemm_7min`, the runner has two separate responsibilities:
- the top-level `RUN_TESTS` list decides whether this benchmark path is active at all
- the lower `if "dense_gemm_7min" in RUN_TESTS:` block defines how to import, execute, and post-process that benchmark

Refocused model for the current session:
- `@cute.kernel` in `dense_gemm_7min.py` is the device-side execution body
- `@cute.jit host_function` is the CuTe host-side builder that assembles MMA, layouts, TMA atoms, multicast parameters, scheduler state, and then launches the kernel
- surrounding Python runner details are secondary unless explicitly requested

## Argument Semantics

- `RUN_TESTS`: dispatch table encoded as a list of string keys
- `M, N, K`: benchmark problem size shared by all branches in this runner
- `warmup_iterations`, `iterations`: performance measurement controls
- `INIT_MODE`, `NORMAL_MEAN`, `NORMAL_STD`: tensor initialization controls forwarded into the kernel runner

CuTe-focused session semantics:
- `tiled_mma`: the logical MMA operator and thread partitioning backbone
- `a_smem_layout` / `b_smem_layout` / `c_smem_layout`: staged shared-memory layout decisions that shape TMA and MMA data movement
- `cluster_layout_vmnk`: the mapping from cluster rank to logical CTA role
- `num_mcast_ctas_a` / `num_mcast_ctas_b`: multicast fanout counts for A and B movement
- `tile_sched_params`: persistent scheduler metadata used to derive grid shape and tile ownership
- `ab_stages`: the number of producer/consumer pipeline stages for A/B movement

Kernel-signature grouping from the debate:
- payload tensors:
  - `mA_mk`, `mB_nk`, `mC_mn`
- transfer machinery:
  - `tma_atom_a`, `tma_atom_b`, `tma_atom_c`
- execution plan:
  - `tiled_mma`, `a_smem_layout`, `b_smem_layout`, `c_smem_layout`, `cluster_layout_vmnk`, `cta_tile_shape_mnk`, `c_layout`, `epi_tiler`, `tile_sched_params`
- execution policy:
  - `num_mcast_ctas_a`, `num_mcast_ctas_b`, `is_a_mcast`, `is_b_mcast`, `num_tma_producer`, `use_2cta_instrs`, `ab_stages`

Current block semantics from the debate:
- `SharedStorage` is control-plane shared memory, not payload tile storage
- `sA`, `sB`, and `sC` are data-plane shared-memory allocations for A, B, and epilogue/C staging
- `tmem` allocation is derived from the accumulator fragment shape rather than hardcoded by hand
- `tCtAcc_fake` is a planning fragment used to size tensor memory before retrieving a real `tmem` pointer
- the kernel uses three distinct pipelines because it has three distinct synchronization boundaries:
  - A/B movement into compute
  - accumulator handoff toward epilogue consumers
  - C/epilogue store path
- `acc_pipeline` consumer count is role-based, not raw-thread-based
- the `acc_pipeline` consumer count is `len(epilogue_warp_id)` scaled by `(2 if use_2cta_instrs else 1)`
- with `epilogue_warp_id = (0, 1, 2, 3)`, the logical consumer count is `4` in 1CTA mode and `8` in 2CTA mode
- the comment `no * cute.arch.WARP_SIZE, must use elect_one in arrive` means accumulator-pipeline arrival is representative and elected, not one arrival per thread
- the three pipelines use different participant-counting models:
  - `ab_pipeline` uses transfer/producer structure
  - `acc_pipeline` uses logical accumulator-consumer roles
  - `c_pipeline` uses CTA-thread participation for the store-side path

## Concrete Examples

- Active selector before change:
  - `"dense_gemm_8_tracer"`
- Active selector after change:
  - `"dense_gemm_7min"`
- Shared benchmark shape in this runner:
  - `M = 8192`, `N = 8192`, `K = 4096`
- Conceptual cluster-role example:
  - if the cluster has two CTA roles along the `v` dimension, a flat cluster rank can map to different logical `v` coordinates
  - rank 0 can act as the leader CTA with `mma_tile_coord_v == 0`
  - rank 1 can act as the non-leader peer CTA with `mma_tile_coord_v != 0`
- Shared-storage grouping example:
  - control plane:
    - `ab_mbar_ptr`, `acc_mbar_ptr`, `tmem_dealloc_mbar_ptr`, `tmem_holding_buf`
  - data plane:
    - `sA`, `sB`, `sC`
- Pipeline grouping example:
  - `ab_pipeline = pipeline.PipelineTmaUmma.create(...)`
  - `acc_pipeline = pipeline.PipelineUmmaAsync.create(...)`
  - `c_pipeline = pipeline.PipelineTmaStore.create(...)`
- `acc_pipeline` consumer-count example:
  - if `use_2cta_instrs == False`, consumer count = `1 * len(epilogue_warp_id) = 4`
  - if `use_2cta_instrs == True`, consumer count = `2 * len(epilogue_warp_id) = 8`
  - this is not multiplied by warp size because arrival uses an elected representative
